# PRD vs Prototype 覆蓋率檢查

你的工作是對照 PRD，系統性地找出 prototype 裡缺少的功能、邏輯與畫面狀態。輸出一份設計師可以直接讀的缺漏清單，以及可以直接交給 AI 補實作的修正指令。

---

## 兩個檢查維度

```
維度 1：功能覆蓋率（本 skill 負責）
  PRD 提到的 feature、AC、business rule，prototype 有沒有實作？

維度 2：狀態覆蓋率（委派給 edge-case-state-mapper）
  每個已實作的功能，必要的 UI 狀態有沒有設計？
  → 不論 PRD 有沒有寫，edge-case-state-mapper 都會從功能性質推導應有狀態
  → 本 skill 拿回推導結果後，比對 prototype 裡哪些有、哪些沒有
```

---

## 執行步驟

### Step 1 — 解析 PRD

從 PRD 提取以下結構，作為後續比對的基準：

```
① Feature 清單：所有功能點（含子功能）
② AC 清單：每個功能的驗收條件（若有）
③ Business Rule 清單：業務邏輯規則（權限、限制、計算規則、例外處理等）
④ 使用者角色：若有多角色，哪些功能只有特定角色能使用
```

若 PRD 缺少上述某項，在報告中標注「PRD 未定義」，不要自行補充。

---

### Step 2 — 掃描 Prototype

從截圖或 HTML 逐頁掃描，建立 prototype 的現況清單：

```
對每個畫面記錄：
- 頁面名稱
- 可見的功能點
- 已設計的 UI 狀態（loading / error / empty 等）
- 可見的 business rule 實作（按鈕 disabled 條件、欄位顯示條件等）
```

---

### Step 3 — 功能覆蓋率比對

對照 Step 1 的 PRD 基準與 Step 2 的 prototype 清單：

```
for 每個 PRD feature / AC / business rule:
  → prototype 有實作？
    ✅ 有：標記覆蓋
    ❌ 沒有：標記缺漏，判斷嚴重度
    ⚠️ 部分：標記部分覆蓋，說明缺少哪個部分
```

嚴重度判斷：

| 嚴重度 | 定義 |
|-------|------|
| 🔴 P0 | 核心流程缺漏，使用者無法完成主要任務 |
| 🟡 P1 | 功能存在但不完整，或 business rule 未實作導致邏輯錯誤 |
| 🟢 P2 | 邊緣功能缺漏，不影響主流程 |

---

### Step 4 — 狀態覆蓋率（委派 edge-case-state-mapper）

**不在這裡自行推導狀態。** 改為：

1. 從 Step 2 的 prototype 清單，整理出每個已實作功能的 happy path 描述
2. 將這份清單交給 `edge-case-state-mapper` 執行，取得完整的應有狀態清單
3. 拿 `edge-case-state-mapper` 的輸出，對照 Step 2 的 prototype 現況，做比對：

```
for 每個 edge-case-state-mapper 推導出的狀態:
  → prototype 已有設計？
    ✅ 有：標記覆蓋
    ❌ 沒有：標記缺漏，沿用 edge-case-state-mapper 的 Priority 分級
```

**委派格式**（傳給 edge-case-state-mapper 的輸入）：

```
以下是這個 prototype 的功能 happy path 清單，
請依序推導每個功能應有的 UI 狀態（六大類別 + 邊界條件）：

功能 1：[名稱]
Happy Path：[描述主流程]

功能 2：[名稱]
Happy Path：[描述主流程]

[以此類推]
```

## 輸出原則

1. **功能維度靠 PRD，狀態維度靠 edge-case-state-mapper**：兩個來源分工明確，不要在本 skill 自行推導狀態規則
2. **PRD 來源要標注**：每個功能缺漏都要說明對應 PRD 哪個部分
3. **已覆蓋的也要列**：讓設計師對整體覆蓋狀況有完整感
4. **PRD 模糊的不要硬判斷**：寫進「需與 PO 確認」，不要自行推斷為缺漏或已覆蓋
5. **AI 修正指令要具體**：給位置、觸發條件、實作描述，AI 不需要再猜
